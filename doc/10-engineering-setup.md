# Engineering Setup

## Prerequisites

- Node.js 20.11+
- npm 10+
- Docker Desktop

## Services

The local Docker Compose stack runs:

- PostgreSQL 16 with PostGIS
- Redis 7

Run:

```bash
docker compose up -d
```

## Database

Prisma schema lives at:

```text
apps/api/prisma/schema.prisma
```

Commands:

```bash
npm run db:generate
npm run db:migrate
npm run db:seed
```

The first migration enables:

- `postgis`
- `pg_trgm`

These support radius search and trigram matching in later public discovery phases.

## API

The API uses route namespaces:

- `/api/public/*`
- `/api/admin/platform/*`
- `/api/admin/tenant/*`
- `/api/admin/auth/*`

OpenAPI is generated by Nest Swagger at:

```text
http://localhost:4000/api/docs
```

Admin auth endpoints:

- `POST /api/admin/auth/login`
- `POST /api/admin/auth/password-reset/request`
- `POST /api/admin/auth/password-reset/confirm`
- `POST /api/admin/auth/logout`
- `GET /api/admin/auth/me`

Login sets an HTTP-only admin session cookie and returns a CSRF token. Unsafe authenticated admin requests must send the token in the `x-csrf-token` header; the same value is also set in the readable `XSRF-TOKEN` cookie for browser clients.
Password reset request responses are intentionally generic to avoid account enumeration. Confirming a valid reset token updates the password, marks the token used, clears failed login state, and deletes old sessions.

Initial platform admin endpoints:

- `GET /api/admin/platform/categories`

Tenant activation enforces the approved MVP requirements: name, primary category, timezone, active location, active service, active expert, active expert-service assignment, and active expert availability.

The first tenant admin endpoint creates an owner membership and returns a temporary password. Email delivery of reset links is intentionally deferred to the notification/password-reset slice.

The customer anonymization endpoint requires either `customerId` or E.164 `phone`, optionally scoped by `tenantId`. It masks customer records, booking customer snapshots, notification recipients/payloads, booking attempts, and practical PII inside audit JSON while preserving operational booking records.

Tenant admin routes require tenant membership and reject cross-tenant access in the guard before controller logic runs.
Booking status changes are limited to the approved MVP transitions from `confirmed` to `cancelled`, `completed`, or `no_show`.
Tenant logo and expert photo uploads are multipart requests using field name `file`. Only JPEG, PNG, WebP, and GIF images are accepted, and max size is read from `UPLOAD_MAX_IMAGE_BYTES`. Uploads are written to S3-compatible storage and recorded in `uploaded_assets`.

## Frontend

The customer web app starts at:

```text
http://localhost:3000
```

Admin screens will live under route groups in the same Next.js app unless requirements later justify a separate app.

Initial customer web routes:

- `/`: search-led home page
- `/search`: tenant discovery results
- `/tenants/:slug`: tenant detail, services, experts, slots, and booking form
- `/booking-lookup`: reference plus phone booking lookup

Client-side API calls use `NEXT_PUBLIC_API_APP_URL` when set, then `API_APP_URL`, then `http://localhost:4000`.

Environment files:

- Root `.env` is retained as a local monorepo convenience file.
- API deployment/runtime variables should come from `apps/api/.env` or the deployment provider's API project settings.
- Web deployment/runtime variables should come from `apps/web/.env` or the deployment provider's web project settings.
- The web app should only receive browser-safe values, currently `NEXT_PUBLIC_API_APP_URL`.

Initial admin web routes:

- `/admin/login`
- `/admin/forgot-password`
- `/admin/reset-password?token=...`
- `/admin/platform`
- `/admin/platform/bookings`
- `/admin/platform/categories`
- `/admin/platform/logs`
- `/admin/platform/tenants/new`
- `/admin/platform/tenants/:id`
- `/admin/tenant`
- `/admin/tenant/:tenantId`

Admin web calls use cookie sessions and send the readable `XSRF-TOKEN` cookie value as the `x-csrf-token` header for mutating requests.
Admin password reset screens call the existing generic request endpoint and token confirm endpoint; reset links generated by the API land on `/admin/reset-password?token=...`.
Platform admin booking lookup supports tenant, date, status, reference, customer, service, and expert filters, and platform admins can cancel confirmed bookings with an audit log and customer cancellation notification.
The shared admin sidebar includes logout. Tenant workspace now supports primary location editing, tenant logo upload, expert photo upload, service/expert creation, expert-service assignment, recurring availability rule creation/archive, availability exception creation/deletion, booking filters, and confirmed booking transitions to cancelled, completed, or no-show.

Initial public discovery endpoints:

- `GET /api/public/categories`
- `GET /api/public/reverse-geocode?latitude=...&longitude=...`
- `GET /api/public/tenants`
- `GET /api/public/tenants/:slug`
- `GET /api/public/tenants/:slug/services`
- `GET /api/public/tenants/:slug/experts`
- `GET /api/public/tenants/:slug/available-slots?serviceId=...&date=YYYY-MM-DD&expertId=...`
- `POST /api/public/tenants/:slug/bookings`
- `GET /api/public/bookings/lookup?reference=...&phone=...`

Tenant location geocoding uses the configured `GOOGLE_MAPS_API_KEY`, is protected by tenant admin membership, and returns formatted address plus latitude/longitude and common address components for the primary location form.

Public tenant endpoints only return `active` tenants. Inactive and suspended tenants return `404` from direct public slug routes and are excluded from search. Search supports keyword, category slug, locality, browser geolocation coordinates, radius in kilometers, and result limit. The home page and `/search` UI include radius selection. Their locality controls include a locate icon that captures browser coordinates, reverse-geocodes them into a readable locality label, and waits for the user to explicitly submit Search before loading tenants. Search ranking uses PostgreSQL `pg_trgm` similarity for tenant/service text matching and PostGIS geography distance filtering when coordinates are provided. The `20260514120000_search_indexes` migration adds trigram and geography indexes for this path.

Available-slot lookup uses the tenant timezone, fixed 15-minute increments, 2-hour minimum notice, and a 30-day advance booking window. Override exceptions replace recurring rules for that date; block exceptions remove availability.

Public booking creation requires customer name, E.164 phone, service, selected UTC slot start, and consent. Bookings are instantly confirmed, references use `{TENANT_PREFIX}-{4_CHAR_CODE}`, and database uniqueness protects against double booking races.

Rate limiting:

- `GET /api/public/tenants`: `RATE_LIMIT_PUBLIC_SEARCH_PER_MINUTE`
- `GET /api/public/tenants/:slug/available-slots`: `RATE_LIMIT_PUBLIC_SLOT_LOOKUP_PER_MINUTE`
- `POST /api/public/tenants/:slug/bookings`: `RATE_LIMIT_PUBLIC_BOOKING_CREATE_PER_MINUTE`
- `GET /api/public/bookings/lookup`: `RATE_LIMIT_PUBLIC_BOOKING_LOOKUP_PER_MINUTE`
- `POST /api/admin/auth/login`: `RATE_LIMIT_ADMIN_LOGIN_PER_WINDOW`

Limits are read from environment variables with local defaults from config validation. The current MVP guard is process-local and can be swapped for Redis-backed distributed counters without changing controller annotations.

Notification foundation:

- Booking creation writes queued notification logs for customer confirmation when email is provided.
- Booking creation writes queued notification logs for tenant owner/admin alerts.
- Tenant booking cancellation writes a queued customer cancellation notification when email is available.
- Queue enqueue uses BullMQ with three attempts and exponential backoff.
- If Redis enqueue fails, the notification log is marked `failed`; booking/cancellation still succeeds.
- `NotificationService.processEmailNotification(logId)` contains the Mailtrap send/update behavior for the worker slice.

Notification worker commands:

```bash
npm --workspace apps/api run build
npm --workspace apps/api run worker
```

Reminder enqueue command:

```bash
npm --workspace apps/api run build
npm --workspace apps/api run reminders:enqueue
```

The reminder command scans confirmed bookings starting roughly 24 hours from the run time and enqueues `booking_reminder_customer` once per booking.

Backend test coverage:

- Scheduling unit tests cover recurring availability, overrides, blocks, and confirmed booking removal.
- Booking unit tests cover any-expert assignment, serializable transaction conflict handling, and rejected attempt logging.
- Public controller flow tests cover the anonymous discovery, slot lookup, booking creation, and booking lookup contract.
- Public search unit tests cover database-ranked search hydration and empty result behavior.
- Platform admin unit tests cover booking lookup filters and platform cancellation side effects.
- Tenant admin unit tests cover Google geocoding result mapping and missing-key handling.
- Auth unit tests cover token hashing plus platform and tenant authorization boundaries.
- Upload and rate-limit unit tests cover image validation and configured guard behavior.
